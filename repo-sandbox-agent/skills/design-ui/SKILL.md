# Design UI Skill

## Operating Modes

### MODE A: Blueprint Mode (called from architect-website pipeline)
Activated when architect provides a complete **Site Blueprint** JSON object.
This mode builds every component using the blueprint's real content — no placeholders ever.

#### Blueprint Mode Steps

**Step 1 — Receive Blueprint**
The architect passes the complete Site Blueprint from Phase 5 of `architect-website`.
Confirm receipt:
```
[uiux-designer] Blueprint received. Theme: <theme> | Zones: 6
Beginning component implementation...
```

**Step 2 — Apply CSS Variables**
Prepend to `generated-site/src/app/globals.css`:
```css
/* === THEME: <theme> — generated by architect-website === */
:root {
  --color-primary:    <palette.primary>;
  --color-secondary:  <palette.secondary>;
  --color-accent:     <palette.accent>;
  --color-bg:         <palette.background>;
  --color-surface:    <palette.surface>;
  --color-text:       <palette.text>;
  --color-text-muted: <palette.text_muted>;
  --font-display:     '<fonts.display.family>', sans-serif;
  --font-body:        '<fonts.body.family>', sans-serif;
}
<all animation CSS from generate-images output>
```

**Step 3 — Inject Fonts in layout.tsx**
Add Google Fonts `<link>` tags in `generated-site/src/app/layout.tsx` `<head>`.

**Step 4 — Build Each Zone Component**
Follow the component specs in architect-website SKILL.md §Phase 6b through §Phase 6h.
Announce each as written:
```
[uiux-designer] ✓ Navbar.tsx written
[uiux-designer] ✓ Hero.tsx written
[uiux-designer] ✓ Card.tsx written
[uiux-designer] ✓ FeatureStrip.tsx written
[uiux-designer] ✓ CTABanner.tsx written
[uiux-designer] ✓ Footer.tsx written
[uiux-designer] ✓ page.tsx assembled
```

**Step 5 — Quality Check**
Before returning to architect:
- [ ] Every `<img>` has a real `src` and `alt`
- [ ] No hardcoded hex colors — all use `var(--color-*)` 
- [ ] All animations have `prefers-reduced-motion` override
- [ ] `page.tsx` imports all 6 zone components
- [ ] No TypeScript errors

### MODE B: Canvas Mode (user explicitly requests visual editor)
Activated ONLY when user says: "open visual editor", "edit visually", "canvas editor",
"hand draw", "use the canvas", "draw the UI", or similar explicit visual editor request.

**CRITICAL**: Do NOT activate Canvas Mode during website generation. Do NOT open :3000 unless user explicitly asks.

#### Canvas Mode Steps

**Step 1: Route Announcement**
```
[orchestrator → code-editor → uiux-designer] Visual editor requested.
Activating canvas-editor hand-tracking interface...
```

**Step 2: Activate canvas-editor**
The canvas-editor Next.js app must be running at `http://localhost:3000`.
Call `launch_frontend('http://localhost:3000')` to start and open it.

**Step 3: Receive Canvas Data**
The `GitAgentRunner` component in canvas-editor sends canvas shape data:
- Shapes array with bounding boxes, types, and text labels
- Natural language instruction (if provided in the runner)

**Step 4: Build Components from Canvas Shapes**
Parse geometric hierarchy from shapes:
- Top 15%: Navbar
- Next 30%: Hero
- Middle 40%: Cards grid
- Bottom 15%: Footer

Generate Next.js/Tailwind components, show diff, run guardrails, sync.

**Step 5: Memory Update**
```markdown
## Last UI Edit
- Timestamp: <ISO date>
- Mode: Canvas
- Components: <list>
- Files: <list>
- Status: synced
```

## Canvas-Editor Reference
| Component | Role |
|---|---|
| `HandTracker.tsx` | MediaPipe webcam → cursor + pinch detection |
| `Tldraw` (dynamic) | Drawing canvas driven by hand cursor |
| `GitAgentRunner.tsx` | Agent status panel, sends shapes, displays log |
| `/api/sync` (POST) | Writes generated code to `../generated-site/` |
